# Testing Analysis Results and Generating Requirements

After the platform is running (containers up, API responding), you can test indexing and requirements generation using either the CLI or the API.

## 1. Verify Everything is Running

```bash
podman ps | grep codeintel
```

All containers (`postgres`, `redis`, `ollama`, `worker`, `api`) should be `Up`.

## 2. Index a Git Repository (CLI)

Use the robust CLI to clone and index a public repo:

```bash
uv run python -m src analyze https://github.com/octocat/Hello-World.git --depth 1
```

Expected output:
```
Cloning Git repository: https://github.com/octocat/Hello-World.git
Cloned to /tmp/codeintel_XXXXXX
Analyzing /tmp/codeintel_XXXXXX...
Indexed with version 1775395679
Cleaned up temporary clone.
```

## 3. Query Analysis Results (API)

After indexing, you can query the API for dead code, call graph, etc.

### 3.1 Dead Code

```bash
curl -X POST http://localhost:8000/query \
  -H "Content-Type: application/json" \
  -d '{"rule": "dead_code"}'
```

You'll get a JSON list of all functions that are never called. For a small repo like Hello-World, you might see something like:

```json
{
  "result": [
    {"symbol_id": "function:hello", "name": "hello", "kind": "function", "file": "/tmp/codeintel_xxx/.../hello.py"}
  ]
}
```

### 3.2 Call Graph (Transitive Calls)

```bash
curl -X POST http://localhost:8000/query \
  -H "Content-Type: application/json" \
  -d '{"rule": "transitive_calls"}'
```

Returns all caller–callee pairs.

### 3.3 Impact Analysis

```bash
curl -X POST http://localhost:8000/query \
  -H "Content-Type: application/json" \
  -d '{"rule": "impact", "symbol": "function:hello"}'
```

Returns callers up to depth 3.

## 4. Generate Requirements (API)

Before generating requirements, ensure Ollama has a model pulled:

```bash
podman exec -it codeintel-ollama ollama pull phi3:mini
```

Then call the requirements endpoint:

```bash
curl -X POST http://localhost:8000/requirements
```

You'll receive a JSON object with an `epic`, `feature`, `user_story`, `acceptance_criteria`, and `tasks` generated by the LLM based on the indexed code.

Example response (abbreviated):
```json
{
  "requirements": "{\n  \"epic\": \"Modernize Hello-World Application\",\n  \"feature\": \"Greeting Service\",\n  \"user_story\": \"As a user, I want to receive a personalized greeting so that I feel welcomed.\",\n  \"acceptance_criteria\": [\"Greeting includes the user's name\", \"Response time < 100ms\"],\n  \"tasks\": [\"Implement hello() function\", \"Add input validation\"]\n}"
}
```

## 5. Test with a Local Repository (Optional)

If you have a local Python project, index it directly:

```bash
uv run python -m src analyze /path/to/your/project
```

Then query and generate requirements as above.

## 6. Monitor Logs for Debugging

- **API logs**: `podman logs codeintel-api --tail 50`
- **Worker logs**: `podman logs codeintel-worker --tail 50`
- **Database**: `podman exec -it codeintel-postgres psql -U postgres -d codeintel -c "SELECT COUNT(*) FROM facts;"`

## 7. Expected Success Criteria

| Test | Expected Result |
|------|----------------|
| Indexing completes without errors | Exit code 0, no tracebacks |
| `dead_code` query returns a JSON list (possibly empty) | HTTP 200, valid JSON |
| `requirements` endpoint returns structured JSON | HTTP 200, contains `epic`, `user_story`, etc. |
| Database contains facts | `SELECT COUNT(*) FROM facts` > 0 |

If any step fails, check the logs and ensure all containers are healthy. The platform is now fully functional for analysis and forward‑engineering requirements.
